# Введение

## Цель работы  

Приобретение практических навыков по настройке динамического распределения IP-адресов посредством протокола DHCP (Dynamic Host Configuration Protocol) в локальной сети.

# Ход выполнения

В Packet Tracer был открыт существующий проект с реализованной сетевой инфраструктурой, включающей маршрутизатор, несколько коммутаторов, серверы и оконечные устройства, распределённые по различным подсетям. В топологии присутствуют сегменты сети, объединённые через центральный маршрутизатор, обеспечивающий межсетевое взаимодействие.

## Добавление и настройка DNS-сервера

В рабочую область был добавлен сервер **Server-PT (dns-ngaforov)** и подключён к коммутатору **msk-donskaya-ngaforov-sw-3** через интерфейс **FastEthernet0/2**. Порт на коммутаторе был активирован.

Далее в настройках сервера на вкладке **Desktop → IP Configuration** был задан статический IP-адрес:
- IP-адрес: **10.128.0.5**
- Маска подсети: **255.255.255.0**
- Шлюз по умолчанию: **10.128.0.1**

![Настройка IP-адреса DNS-сервера](Screenshot_2.png){ #fig:009 width=70% }

## Настройка службы DNS

На сервере была открыта вкладка **Services → DNS**, после чего служба DNS была активирована (переключена в состояние **On**).

Были добавлены ресурсные записи типа **A (A Record)**, обеспечивающие разрешение доменных имён в IP-адреса:

- **dns.donskaya.rudn.edu → 10.128.0.5**
- **file.donskaya.rudn.edu → 10.128.0.3**
- **mail.donskaya.rudn.edu → 10.128.0.4**
- **www.donskaya.rudn.edu → 10.128.0.2**

Каждая запись добавлялась через поля **Name** и **Address** с последующим нажатием кнопки **Add**.

![Настройка DNS-записей (A Record)](Screenshot_3.png){ #fig:010 width=75% }

Таким образом, была реализована служба доменных имён, позволяющая обращаться к серверам по символьным именам.

## Исследование работы DHCP в режиме Simulation

После настройки DHCP-сервера сеть была переведена в режим **Simulation** для детального анализа процесса получения IP-адреса клиентом.

В панели событий (**Event List**) отображаются пакеты DHCP, проходящие через сеть. При запуске симуляции наблюдается последовательный обмен сообщениями между клиентом и DHCP-сервером.

![Передача DHCP-сообщений в режиме Simulation](Screenshot_10.png){ #fig:017 width=80% }

Процесс получения IP-адреса по протоколу DHCP состоит из четырёх основных этапов (модель DORA):

### DHCP Discover

Клиент, не имеющий IP-адреса, отправляет широковещательный пакет **DHCP Discover**:
- Source IP: **0.0.0.0**
- Destination IP: **255.255.255.255**
- UDP порт: **68 → 67**

В пакете указывается MAC-адрес клиента, по которому сервер сможет идентифицировать устройство.

![Структура DHCP Discover](Screenshot_11.png){ #fig:018 width=85% }

На этом этапе:
- поле **Client Address** равно 0.0.0.0  
- поле **Your Client Address** также пустое  
- указан **Client Hardware Address (MAC)**  

### DHCP Offer

DHCP-сервер отвечает клиенту сообщением **DHCP Offer**, предлагая свободный IP-адрес.

Основные параметры:
- предлагаемый IP-адрес (например, **10.128.4.30**)  
- адрес DHCP-сервера (**10.128.4.1**)  
- дополнительные параметры (DNS, маска и др.)

![Структура DHCP Offer](Screenshot_12.png){ #fig:019 width=85% }

В пакете:
- поле **Your Client Address** содержит предлагаемый IP  
- указывается DNS-сервер (**10.128.0.5**)  

### DHCP Request

Клиент принимает предложение и отправляет сообщение **DHCP Request**, подтверждая выбор IP-адреса.

Этот пакет также передаётся широковещательно, чтобы:
- уведомить все DHCP-серверы  
- подтвердить выбор конкретного предложения  

![Структура DHCP Request](Screenshot_13.png){ #fig:020 width=85% }

### DHCP Acknowledgment (ACK)

Сервер завершает процесс, отправляя **DHCP ACK**, подтверждая выдачу адреса клиенту.

![Структура DHCP ACK](Screenshot_14.png){ #fig:021 width=85% }

В данном пакете:
- IP-адрес окончательно закрепляется за клиентом  
- передаются параметры:
  - шлюз по умолчанию  
  - DNS-сервер (**10.128.0.5**)  
  - маска подсети  

## Анализ структуры DHCP-пакетов

DHCP работает поверх протокола **UDP**:
- клиент использует порт **68**  
- сервер — порт **67**

В заголовке DHCP присутствуют ключевые поля:

- **OP** — тип сообщения (запрос или ответ)  
- **Transaction ID** — идентификатор сессии  
- **Client Hardware Address** — MAC-адрес клиента  
- **Your Client Address** — выделяемый IP-адрес  
- **Server Address** — адрес DHCP-сервера  

Дополнительно используются **DHCP Options**, где передаются параметры:
- DNS-сервер (Option 6)  
- доменное имя  
- время аренды (lease time)

## Контрольные вопросы

### 1. За что отвечает протокол DHCP?

**DHCP (Dynamic Host Configuration Protocol)** — это протокол автоматической настройки сетевых параметров узлов.  
Он отвечает за выдачу устройствам IP-адресов и других параметров, необходимых для работы в сети.

С помощью DHCP клиент автоматически получает:
- IP-адрес  
- маску подсети  
- шлюз по умолчанию  
- адрес DNS-сервера  

Это позволяет упростить администрирование сети и избежать ошибок при ручной настройке.

### 2. Какие типы DHCP-сообщений передаются по сети?

В процессе работы DHCP используется последовательность сообщений, известная как модель **DORA**:

- **DHCP Discover** — широковещательный запрос клиента на получение IP-адреса  
- **DHCP Offer** — ответ сервера с предложением IP-адреса  
- **DHCP Request** — запрос клиента на подтверждение выбранного адреса  
- **DHCP ACK (Acknowledgment)** — подтверждение сервера о выдаче адреса  

Дополнительно могут использоваться:
- **DHCP NAK** — отказ в выдаче адреса  
- **DHCP Release** — освобождение адреса клиентом  
- **DHCP Inform** — запрос дополнительных параметров  

### 3. Какие параметры могут быть переданы в сообщениях DHCP?

DHCP-сервер может передавать клиенту различные параметры конфигурации сети:

- IP-адрес устройства  
- маска подсети  
- шлюз по умолчанию  
- адрес DNS-сервера  
- доменное имя  
- время аренды IP-адреса (lease time)  
- адреса дополнительных серверов (например, NTP, TFTP)  

Эти параметры передаются через поле **DHCP Options**.

### 4. Что такое DNS?

**DNS (Domain Name System)** — это система доменных имён, предназначенная для преобразования символьных имён (например, *www.example.com*) в IP-адреса.

DNS позволяет пользователям обращаться к ресурсам сети по удобным именам вместо числовых адресов.  
Работает по принципу клиент-сервер и использует распределённую иерархическую структуру.

### 5. Какие типы записи описания ресурсов есть в DNS и для чего они используются?

В DNS используются различные типы ресурсных записей:

- **A (Address Record)** — сопоставляет доменное имя с IPv4-адресом  
- **AAAA** — сопоставляет имя с IPv6-адресом  
- **CNAME (Canonical Name)** — создаёт псевдоним для другого доменного имени  
- **MX (Mail Exchange)** — указывает почтовые серверы домена  
- **NS (Name Server)** — определяет DNS-серверы, обслуживающие домен  
- **PTR (Pointer Record)** — используется для обратного разрешения (IP → имя)  
- **SOA (Start of Authority)** — содержит основную информацию о зоне DNS  

Эти записи обеспечивают работу системы доменных имён и позволяют эффективно управлять сетевыми ресурсами.
